# 성남시 전기차 보조금 공고 변화 알림 — 설계 문서

- 작성일: 2026-06-18
- 상태: 설계 확정 (구현 전)
- 레포: `~/dev/github/ev-subsidy-watch` (개인 github.com, public)

## 1. 목적

[무공해차 통합누리집 구매보조금 지급현황 페이지](https://ev.or.kr/nportal/buySupprt/initSubsidyPaymentCheckAction.do)의
**성남시 행**을 주기적으로 읽어, **전기차 보조금(특히 하반기 추경) 공고가 갱신되면 텔레그램으로 알림**을 받는다.

> 알림 채널 결정: 사내 Slack은 (1) admin 정책상 인커밍 webhook 생성이 막힐 수 있고 (2) 개인 프로젝트를 사내 워크스페이스로 흘리고 사내 webhook을 개인 public 레포 Secret에 저장하는 것이 거버넌스상 부적절하여 제외. 텔레그램은 봇 토큰이 만료·리프레시가 없고 GitHub Actions에서 바로 전송 가능해 채택. (카카오톡 "나에게 보내기"는 OAuth 토큰 6시간 만료·리프레시 관리 부담으로 제외.)

현재 성남시 비고란은 "공고 마감 / 추경예산 확보 후 공고 및 접수 예정"이며, 추경 공고가 열리면 이 텍스트나 공고파일이 바뀐다.

알림은 두 가지로 보낸다: **(1) 변화 감지** — 15분마다 확인해 공고가 바뀌면 즉시 알림. **(2) 일일 보고서** — 매일 08:00 KST에 현황(잔여대수 등) 요약 전송(하트비트 겸용).

## 2. 핵심 제약 — 페이지가 암호화되어 있음 (스파이크로 검증 완료)

대상 페이지는 `pnp4web`/`penc` 솔루션으로 **본문 HTML과 AJAX 요청·응답이 모두 암호화**되어 있다.

- `curl`로 받은 1.2MB HTML 안에 "성남"이라는 글자조차 없다 (암호화된 상태).
- 조회 시 `initSubsidyPaymentCheckAction.do`로 보내는 POST 응답(785KB)도 평문에 "성남/추경" 없음.
- `getLocalCd1.ajax` 등 JSON 엔드포인트는 **요청 파라미터 자체가 암호화**되어 있다 (예: `post='53mZI6HF8TmwI6kS//PWLf'`).

**결론:** 직접 HTTP/API 호출(접근 B)은 불가능하다. pnp4web 암호화 로직을 통째로 재현해야 하므로 깨지기 쉽다.
→ **실제 브라우저(Playwright/Chromium)가 JS를 실행해 복호화한 DOM을 읽는 방식(접근 A)을 채택한다.** 스파이크에서 end-to-end 동작을 확인했다.

## 3. 모니터링 대상 — 정확한 추출 규칙

조회 결과 표 구조 (10개 컬럼):

```
[0]시도 [1]지역구분 [2]차종구분 [3]공고파일 [4]접수방법 [5]민간공고대수 [6]접수대수 [7]출고대수 [8]출고잔여대수 [9]비고
```

성남시 행 예시 (확인된 실제 값):

```
[0] 경기
[1] 성남시            ← 행 식별자 (셀 텍스트가 정확히 "성남시"인 <tr>을 찾는다)
[2] 전기승용
[3] 본공고1 본공고2 본공고3   → goDownloadFile('2026','4113','A' / 'A02' / 'A03')
[4] *일반: 출고등록순 *우선: 출고등록순
[5] 1949 (0)(0)(0)(1949)      ← 매일 변동, 제외
[6] 1745 (550)(49)(60)(1086)  ← 매일 변동, 제외
[7] 1740 (549)(49)(60)(1082)  ← 매일 변동, 제외
[8] 209 (0)(0)(0)(867)        ← 매일 변동, 제외
[9] ★ 성남시 전기자동차(승용, 화물) 보급사업 공고 마감★ ... 추경예산 확보 후 공고 및 접수 예정 ...  ← 핵심 신호
```

**추출 규칙:**
1. 조회 결과 표에서 `지역구분` 셀의 텍스트가 정확히 `"성남시"`인 `<tr>`를 모두 찾는다. (다른 행은 절대 읽지 않는다.)
2. 각 성남시 행에서 다음만 스냅샷에 포함한다:
   - **[3] 공고파일**: 버튼 라벨 + 각 버튼의 `goDownloadFile(...)` 코드 (예: `A`, `A02`, `A03`). 추경 공고가 추가되면 새 버튼/코드가 생기므로 신호가 된다.
   - **[9] 비고**: 공고문 전체 텍스트 (정규화).
   - (보조) [2] 차종구분, [4] 접수방법 — 안정적이므로 함께 포함해 컨텍스트로 사용.
3. **[5]~[8] 숫자 컬럼은 제외한다** (잔여대수 등은 매일 변동 → 오탐 원인).
4. 성남시 행이 여러 개(예: 승용 + 화물)면 모두 캡처한다. 행 수가 늘어나는 것 자체도 변화로 감지된다.

**탐색 절차 (Playwright):**
`#localDo_cd = "4100"`(경기도) 선택 → 대기 → `#local_cd1 = "4113"`(성남시) 선택 → 대기 → `button:has-text('조회')` 클릭 → 성남시 행이 렌더될 때까지 대기.
- `wait_until="networkidle"`은 pnp4web의 백그라운드 요청 때문에 타임아웃됨 → `domcontentloaded` + 명시적 `wait_for_selector`(성남시 행/비고 텍스트 등장)로 대기한다.

## 4. 아키텍처 & 컴포넌트

개인 github.com의 **public 레포 1개**. 핵심 원칙: **`check.py`는 호스트에 무관하게 동작**하고, GitHub Actions는 그저 기본 실행 호스트다 (R1 대비, 9절).

작은 단일 책임 모듈로 분리한다(독립 테스트 가능):

| 모듈 | 역할 | 의존 |
|---|---|---|
| `scraper.py` | `fetch_seongnam_rows() -> list[dict]` : Playwright로 경기도→성남시 조회 후 성남시 행들의 **전체** 데이터 `{차종, 공고파일코드, 접수방법, 민간공고대수, 접수대수, 출고대수, 출고잔여대수, 비고}` 반환 (숫자는 전체/우선순위/법인·기관/택시/일반 분해) | Playwright |
| `state.py` | 직전 스냅샷(`state/seongnam.json`) 로드/저장, 정규화·해시(변화 감지용), 전일 숫자 보관(보고서 증감용), diff 계산 | 표준 라이브러리 |
| `notifier.py` | `send_telegram(text)` : `api.telegram.org/bot<token>/sendMessage` POST (2~3회 재시도) | httpx, `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID` |
| `check.py` | 오케스트레이터. **두 모드**: `--mode change`(변화 감지) / `--mode daily`(일일 보고서). fetch → (change: 해시 비교 → 변화 시 알림) / (daily: 현황+전일 대비 보고서 전송) → 상태 저장 | 위 모듈 |

`check.py` 보조 옵션: `--dry-run`(추출만 출력, 전송·저장 안 함), `--get-chat-id`(getUpdates로 chat_id 출력), `--test-telegram`(테스트 메시지 전송). 14절 설정 가이드에서 사용.

## 5. 데이터 흐름

공통(두 모드): cron → GitHub Actions 트리거 → 레포 체크아웃(직전 상태 포함) → Python + Playwright(chromium) 설치 → `scraper`가 성남시 행 **전체** 추출.

**변화 감지 모드 (`--mode change`, 15분마다)**
1. 추출한 행에서 `비고 + 공고파일`만 정규화·canonical 직렬화 → sha256.
2. 직전 해시와 비교:
   - **같으면** 종료 (아무 동작 없음).
   - **다르면** 텔레그램 알림(이전→현재 diff + 키워드 플래그 + 링크) 전송 → 새 상태 파일을 레포에 **커밋·푸시**(다음 비교 기준).
3. **첫 실행**(상태 없음): 알림 대신 "✅ 감시 시작 — 현재 상태: …" 1회 전송 후 베이스라인 저장.

**일일 보고서 모드 (`--mode daily`, 매일 08:00 KST)**
1. 추출한 행의 현황(공고문 요약 + 잔여대수/접수/출고/공고 숫자 + 공고파일)을 구성.
2. 상태에 보관된 **전일 숫자와 비교해 증감**(예: 잔여 -3)을 덧붙임.
3. 변화 여부와 **무관하게 매일 1회** 보고서 전송 → 오늘 숫자를 `last_report`로 저장.
   - 보고서는 "모니터 살아있음" 하트비트 역할도 한다(끊기면 고장 신호).
   - 보고서 모드는 변화 감지용 `hash`를 건드리지 않는다(두 모드의 상태가 서로 독립).

## 6. 알림 형식 (텔레그램)

공통: `sendMessage`에 `parse_mode`(Markdown 또는 HTML)와 `disable_web_page_preview=true` 사용. 길면 텔레그램 4096자 제한에 맞춰 자른다. 메시지의 모든 시각은 **KST**로 표기.

### 6.1 변화 감지 알림
- 제목: `🔔 성남시 전기차 보조금 공고 변화 감지`
- **키워드 플래그**: 비고/공고파일에서 `추경`, `접수`, `공고`, `마감` 출현 여부 및 "마감" 문구 사라짐 여부.
- **diff**: 이전 비고 → 현재 비고의 변경 부분(unified diff 일부), 공고파일 코드 집합 변화(추가/삭제된 코드).
- 페이지 링크.
- 같은 내용은 새 상태로 저장되므로 반복 알림 없음(스팸 방지).

### 6.2 일일 보고서 (매일 08:00 KST)
- 제목: `📋 성남시 전기차 보조금 일일 현황 (YYYY-MM-DD)`
- **잔여대수**(출고잔여대수): 전체 + 분해(일반 등), **전일 대비 증감** 표기 (예: `전체 209 (일반 867) [전일 -3]`).
- 접수대수 / 출고대수 / 민간공고대수 요약, 공고파일 목록.
- **비고(공고문) 요약**: 현재 공고 상태(예: "공고 마감 / 추경 예정") 앞부분.
- 페이지 링크.
- 변화 유무와 무관하게 매일 전송(하트비트 겸용).

## 7. 상태 저장

`state/seongnam.json`:
```json
{
  "hash": "<sha256: 비고+공고파일 기준>",
  "rows": [ {
    "차종": "전기승용", "공고파일": ["A","A02","A03"], "접수방법": "...", "비고": "...",
    "민간공고대수": {"전체":1949,"우선순위":0,"법인기관":0,"택시":0,"일반":1949},
    "접수대수": {"전체":1745,"...":"..."}, "출고대수": {"...":"..."}, "출고잔여대수": {"전체":209,"일반":867}
  } ],
  "fetched_at": "2026-06-18T08:00:00+09:00",
  "last_report": { "date": "2026-06-17", "출고잔여대수": {"전체":212}, "...": "전일 대비 계산용 숫자" },
  "consecutive_failures": 0
}
```
- 변화 감지 해시: `비고 + 공고파일`만 대상(숫자 제외). 정규화는 모든 공백을 단일 공백으로 축약·trim 후 canonical JSON 직렬화 → sha256.
- `last_report`: 일일 보고서의 전일 대비 증감 계산용(보고서 모드만 갱신). 변화 감지 해시와 독립.
- 상태는 레포에 커밋되어 실행 호스트(Actions/맥)가 바뀌어도 유지된다.

## 8. 에러 처리 — 사일런트 실패 방지

모니터가 조용히 망가지면 알림 누락으로 이어지므로 명시적으로 방어한다.

- **스크랩 실패**(타임아웃 / 성남시 행 미발견 / 차단): **상태 파일을 덮어쓰지 않는다**(베이스라인 보존). `consecutive_failures` 증가.
- "성남시 행을 못 찾음"(셀렉터 실패 = 스크랩 깨짐)과 "내용이 진짜 바뀜"을 구분한다. 표/행 자체가 안 보이면 실패로 간주(변화로 오인 금지).
- **N회 연속 실패 시에만** `⚠️ 체크 실패 N회` 텔레그램 알림 (일시적 오류로 인한 스팸 방지). 기본 N=3.
- 실패 시 **스크린샷 + DOM 일부를 워크플로 아티팩트**로 업로드해 디버깅.
- **텔레그램 전송 실패**: 지수 백오프로 2~3회 재시도.

## 9. 실행 환경

**기본: GitHub Actions** (`.github/workflows/`)
- 두 개의 스케줄(한 워크플로의 cron 2개 + `if`로 모드 분기, 또는 워크플로 2개):
  - **변화 감지**: `cron '*/15 * * * *'` → `python check.py --mode change` (15분 주기는 타임존 무관).
  - **일일 보고서**: `cron '0 23 * * *'` → `python check.py --mode daily`.
- 둘 다 `workflow_dispatch`로 수동 실행 가능.
- 단계: checkout → setup Python(uv) → `playwright install chromium` → `python check.py --mode ...` → 변경/보고 시 `state/` 커밋·푸시.
- `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`는 레포 Secret. 코드엔 비밀 없음.
- public 레포 → Actions 실행 시간 무제한 무료.

> **cron 타임존 주의**: GitHub Actions의 `schedule` cron은 **UTC 고정**(타임존 설정 불가). 한국(KST)은 **서머타임이 없어 UTC+9가 영구 고정**이므로 `0 23 * * *`(UTC)는 *항상* 08:00 KST다 — 드리프트·혼동 없음. YAML에는 `# 23:00 UTC = 08:00 KST` 주석을 단다. 코드/메시지의 모든 시각은 `Asia/Seoul`(KST)로 표기한다. (KST 기준으로 직접 거는 게 꼭 필요하면 launchd 폴백은 로컬 타임존이라 08:00을 그대로 사용 가능.)

**폴백: 본인 맥 launchd** — 동일 `check.py`를 한국 IP·로컬(KST) 타임존에서 실행. R1(아래)이 현실화되면 전환.

## 10. 위험 & 검증 계획

- **R1 (최대): GitHub 러너(미국 Azure IP)가 ev.or.kr에 닿고 pnp4web 복호화까지 통과하는가?** 한국 공공사이트의 해외 IP 차단 가능성.
  - 검증: 구현 직후 첫 `workflow_dispatch`를 `--dry-run`으로 실행해 성남시 비고 텍스트가 출력되는지 확인.
  - 폴백: 막히면 맥 launchd로 전환(코드 변경 없음). 또는 프록시.
- **R2: 헤드리스 탐지** — pnp4web가 headless를 차단할 수 있음. 완화: 실제 UA, locale ko-KR, 필요 시 stealth 옵션/headed-xvfb. (맥 로컬 스파이크에선 headless로 성공.)
- **R3: DOM 셀렉터 취약성** — 정부 사이트 구조 변경. 완화: "성남시" 텍스트 기반 행 탐색(인덱스 의존 최소화) + 실패 알림 + 아티팩트 덤프.

- `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID` (GitHub Secret / 맥 실행 시 env). 발급 방법은 14절 참조.
- 상수: 경기도=`4100`, 성남시=`4113`, 연도=현재년도(기본 2026), 차종 기본값.
- `CONSECUTIVE_FAILURE_THRESHOLD`(기본 3).
- 스케줄: 변화 감지 15분(`*/15 * * * *`), 일일 보고서 08:00 KST(`0 23 * * *` UTC). 워크플로 YAML의 cron 값으로 관리(변경 쉬움).

## 12. 테스트 전략

- **순수 로직 TDD**: `state`의 정규화/해시/diff, 숫자 셀 파싱(전체/우선순위/법인기관/택시/일반), 전일 대비 증감 계산, 키워드 플래그 판정, 메시지 포맷(변화 알림 / 일일 보고서) — 픽스처 기반 단위 테스트.
- **스크래퍼**: 실제 사이트 대상 `python check.py --dry-run` 라이브 스모크(전송·저장 없음). 저장된 HTML 픽스처는 암호화 때문에 무의미하므로 라이브로 검증.

## 13. 비범위 (YAGNI)

- 성남시 외 다른 지자체 모니터링.
- 공고문 hwpx/pdf 파일 내용 파싱(버튼/코드 존재 여부만 감지).

## 14. 텔레그램 설정 가이드 (처음 쓰는 경우)

사용자가 텔레그램을 처음 쓰므로, 구현 중 아래 순서로 함께 진행한다. **1~3은 사용자가 직접**(앱·BotFather 상호작용), **4~5는 구현 측이 헬퍼로 지원**한다.

1. **텔레그램 설치·가입**: 모바일 또는 데스크톱 앱 설치 → 전화번호로 가입.
2. **봇 생성**: 텔레그램에서 `@BotFather` 검색 → `/newbot` → 봇 표시이름과 username(끝이 `bot`) 입력 → **봇 토큰** 발급(`123456789:AAH...` 형식). 이게 `TELEGRAM_BOT_TOKEN`.
3. **봇과 대화 시작**: BotFather가 준 링크 `t.me/<봇username>`을 열어 **`/start`** 를 한 번 누른다. (이걸 해야 봇이 나에게 메시지를 보낼 수 있음.)
4. **chat_id 알아내기**: 구현 측이 헬퍼 제공 — `python check.py --get-chat-id`(또는 별도 스크립트)가 `getUpdates`를 호출해 `chat.id`를 출력. 이게 `TELEGRAM_CHAT_ID`. (대안: `@userinfobot`에게 말 걸어 확인.)
5. **테스트 전송**: `python check.py --test-telegram`으로 "✅ 연결 테스트" 메시지를 보내 휴대폰에서 수신 확인.
6. **시크릿 등록**: 두 값을 GitHub 레포 `Settings → Secrets and variables → Actions`에 `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`로 등록. (맥 실행 시엔 환경변수 또는 `.env` 로컬 파일 — 깃에 커밋 금지.)

구현 계획에는 위 헬퍼(`--get-chat-id`, `--test-telegram`)를 초기 작업에 포함하여, 사용자가 단계마다 동작을 확인할 수 있게 한다.
